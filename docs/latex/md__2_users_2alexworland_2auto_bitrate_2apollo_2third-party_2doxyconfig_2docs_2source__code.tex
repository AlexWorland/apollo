\hypertarget{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md305}{}\doxysubsection{\texorpdfstring{Doxygen Comments}{Doxygen Comments}}\label{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md305}

\begin{DoxyItemize}
\item Use Doxygen comments to document all files, functions, classes, and variables.
\item Inline documenation block should use the following format\+:

\begin{DoxyVerb}///< A brief description of the member.
\end{DoxyVerb}

\item Multi-\/line comments, such as for a documentation block, should use the following format\+:

\begin{DoxyVerb}/**
 * @brief A brief description of the member.
 * More detailed description of the member.
 */
\end{DoxyVerb}

\end{DoxyItemize}\hypertarget{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md306}{}\doxysubsection{\texorpdfstring{Documentation Blocks}{Documentation Blocks}}\label{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md306}
Documentation blocks should be placed above the declaration of the function, class, or variable. Below is an example of a documentation block for the main function.

\begin{DoxyVerb}/**
 * @brief Main application entry point.
 * @param argc The number of arguments.
 * @param argv The arguments.
 * @return The exit code.
 * @examples
 * main(1, const char* args[] = {"hello", "markdown", nullptr});
 * @examples_end
 */
int main(int argc, char *argv[]);
\end{DoxyVerb}


\begin{DoxyNote}{Note}
The @examples and @examples\+\_\+end tags are not standard Doxygen tags. They are custom aliases we have specified to simplify documenting examples. Do not confuse this with the standard @example tag.
\end{DoxyNote}
In some cases, it could be valuable to have slightly different documentation for the definitions, especially when the definition may change depending on the platform or other factors. In such cases, you should put the documentation that is common in the declaration and the definition-\/specific documentation in the definition. Below is an example of how to document the declaration and definition of a function.

\begin{DoxyVerb}// myFile.h
/**
 * @brief A brief description of the function.
 * @param arg1 Describe the first argument.
 * @param arg2 Describe the second argument.
 * @return Describe the result.
 */
int myFunction(int arg1, int arg2);

// myFile.cpp
/**
 * This describes anything which is specific to the implementation of the function.
 */
int myFunction(int arg1, int arg2)
{
    // Implementation
}
\end{DoxyVerb}
\hypertarget{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md307}{}\doxysubsection{\texorpdfstring{File Documentation}{File Documentation}}\label{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md307}
The file documentation block must be placed at the top of the file. If it is not present, Doxygen will ignore the file. Understandably, it is difficult to make a creative description for every file, although it is still required. Below is an example of a file documentation block.

\begin{DoxyVerb}/**
 * @file src/main.cpp
 * @brief Main application entry point.
 */
\end{DoxyVerb}
\hypertarget{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md308}{}\doxysubsection{\texorpdfstring{Inline Documentation Blocks}{Inline Documentation Blocks}}\label{md__2_users_2alexworland_2auto_bitrate_2apollo_2third-party_2doxyconfig_2docs_2source__code_autotoc_md308}
Inline comments can be used to describe enum values, variables, and other code constructs. To document the members of a file, struct, union, class, or enum, it is sometimes desired to place the documentation block after the member instead of before. For this purpose you have to put an additional {\ttfamily \texorpdfstring{$<$}{<}} marker in the comment block. Below is an example of an inline comment for an enum value.

\begin{DoxyVerb}enum class MyEnum
{
   FIRST_VALUE,  ///< A brief description of the FIRST_VALUE
   SECOND_VALUE  ///< A brief description of the SECOND_VALUE
};
\end{DoxyVerb}

We have defined some custom aliases to simplify documenting examples.


\begin{DoxyItemize}
\item @examples -\/ Start of an example block. This will format the following text as {\ttfamily cpp}.
\item @examples\+\_\+end -\/ End of an example block.
\item @fa\+\_\+icon\{font awsome definition\} -\/ Insert a Font Awesome icon. Give the icon style and name as arguments. e.\+g. @fa\+\_\+icon\{fa-\/solid fa-\/check\}.
\item Admonitions
\begin{DoxyItemize}
\item @admonition\{Custom Title \texorpdfstring{$\vert$}{|} Content \} @admonition\{The one with the custom titles \texorpdfstring{$\vert$}{|} It’s got a certain charm to it.\}
\item @attention\{Content\} \begin{DoxyAttention}{Attention}
\{The sun causes global warming.\}
\end{DoxyAttention}

\item @caution\{Content\} @caution\{Cliff ahead\+: Don’t drive off it.\}
\item @danger\{Content\} @danger\{Mad scientist at work!\}
\item @error\{Content\} @error\{Does not compute.\}
\item @hint\{Content\} @hint\{Insulators insulate, until they are subject to \+\_\+\+\_\+\+\_\+\+\_\+\+\_\+\+\_\+ voltage.\}
\item @important\{Content\} \begin{DoxyImportant}{Important}

\end{DoxyImportant}

\item @note\{Content\} \begin{DoxyNote}{Note}
